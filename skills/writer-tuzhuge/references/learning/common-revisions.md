## 修改类型统计

| 类型 | 预估频率 | 典型反馈 | 预防措施 |
|-----|---------|---------|---------|
| 开头太平淡 | 35% | "没有吸引力""不想继续看" | 默认使用悬念式/痛点式开头 |
| 内容太官方 | 25% | "像说明书""没有人味" | 增加个人体验描述，加入吐槽 |
| 结构不清晰 | 20% | "看不出重点""逻辑混乱" | 使用01/02/03分段+小标题 |
| 篇幅过长 | 10% | "太啰嗦""看不完" | 控制在用户指定字数±10% |
| 篇幅过短 | 5% | "太简单""不够深入" | 增加案例和细节展开 |
| 缺少案例 | 5% | "太抽象""没有说服力" | 每个功能/观点配具体场景 |

## 预防性检查清单

生成文章后、交付用户前，自动检查以下项目：

### 开头检查

- [ ] 开头是否有悬念/痛点/故事？
- [ ] 前3句是否能吸引读者继续看？
- [ ] 是否避免了"随着...的发展"等套话开头？

### 内容检查

- [ ] 是否包含至少1处个人体验描述？
- [ ] 是否有"我觉得""说实话"等主观表达？
- [ ] 是否有吐槽或调侃增加人味？
- [ ] 每个功能/观点是否有具体案例？

### 结构检查

- [ ] 是否使用01/02/03分段？（注意：教程类文章改用加粗标题，不用编号）
- [ ] 每段是否有清晰的主题？
- [ ] 段落之间是否有逻辑递进？
- [ ] 是否避免了"首先、其次、最后"连续出现？

### 篇幅检查

- [ ] 字数是否在用户指定范围内？
- [ ] 是否有明显的水字数段落？
- [ ] 是否有可以删除的冗余内容？

### 风格检查

- [ ] 是否符合用户选定的风格？
- [ ] 标志性表达是否到位？
- [ ] 结尾是否有签名？（注意：签名不是必须的）

### 类比检查

- [ ] 类比是否自然贴切？不贴切的宁可不要
- [ ] 是否有过度类比？（一篇文章中类比不宜过多，点到为止）
- [ ] 类比是否会让文章显得刻意？

### 过渡句检查

- [ ] 段落之间的过渡是否自然口语化？
- [ ] 是否使用了"顺带说一下""最后再唠叨几句"等口语过渡？（优于"再说""写在最后"）
- [ ] 结尾是否附上相关下载/体验链接？

### 读者友好检查

- [ ] 正文中是否暴露了 .md / .jsonl 等技术文件名？（面向普通用户时必须去掉）
- [ ] 讲自己的工作流时是否用了第一人称"我"？（不要用"用户"）
- [ ] 连贯对比句是否用逗号而非破折号？（更顺畅）

## 常见问题修复指南

### 问题1：开头太平淡

**症状**：
- 直接介绍产品/话题
- 使用"今天我们来聊聊..."
- 没有悬念或吸引力

**修复方法**：
1. 改用悬念反转式开头
2. 或改用痛点共鸣式开头
3. 或改用故事引入式开头

**修复示例**：

❌ 原开头：
> 今天我们来聊聊Lovart这款AI设计工具。

✅ 修复后：
> 一直不敢写Lovart。
>
> 因为它实在太火了。小红书、即刻、X，随便一搜，铺天盖地都是"Lovart太强了"。

---

### 问题2：内容太官方

**症状**：
- 像产品说明书
- 没有个人观点
- 缺少情感表达

**修复方法**：
1. 在关键位置加入"我觉得""说实话"
2. 加入个人使用体验
3. 加入吐槽或调侃

**修复示例**：

❌ 原文：
> 该功能可以帮助用户快速生成设计稿。

✅ 修复后：
> 说实话，这个功能我第一次用的时候就惊了。以前做个海报要半天，现在几分钟就搞定了。

---

### 问题3：结构不清晰

**症状**：
- 没有明确分段
- 逻辑跳跃
- 读者不知道在讲什么

**修复方法**：
1. 使用01/02/03分段
2. 每段开头点明主题
3. 段落之间加过渡

**修复示例**：

❌ 原结构：
> 这个产品有很多功能。首先是A功能，其次是B功能，最后是C功能。

✅ 修复后：
> 01
>
> 先说最让我惊艳的功能：A。
>
> [展开描述]
>
> 02
>
> 但A还不是最厉害的。真正让我"哇"出声的，是B。
>
> [展开描述]

---

### 问题4：篇幅问题

**篇幅过长的修复**：
1. 删除重复表达
2. 合并相似段落
3. 删除与主题无关的内容
4. 精简案例描述

**篇幅过短的修复**：
1. 增加具体案例
2. 展开细节描述
3. 加入对比分析
4. 补充个人体验
